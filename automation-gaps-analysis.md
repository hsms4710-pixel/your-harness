# Harness Skills "自动化"不足分析 - 从用户故事出发

## 核心问题

当前 harness skills 过度强调"自动化"，但忽略了**需求澄清**和**用户交互**的重要性。

**自动化 ≠ 不需要交流**

真正的自动化应该是：
- 自动执行**机械性**任务
- 对于**决策性**任务，需要与用户交流澄清需求

## 4 个用户故事的"自动化"不足

### 用户故事 1：新项目开发者 - 缺少需求澄清

**当前流程（过于自动化）：**
```
用户: "帮我生成 Harness"
  ↓
自动识别: Greenfield 模式
  ↓
自动询问: "请告诉我技术栈"
  ↓
用户填写技术栈
  ↓
自动生成标准系统
```

**问题：缺少需求澄清**

| 缺失的需求 | 影响 | 应该澄清的问题 |
|-----------|------|---------------|
| 项目规模 | 生成的系统复杂度不匹配 | "这是个人项目还是团队项目？团队多大？" |
| 质量要求 | 门禁阈值不匹配 | "对代码质量的要求是严格还是宽松？" |
| CI/CD 集成 | 生成的 CI 配置不匹配 | "使用什么 CI 平台？GitHub Actions/GitLab CI/Jenkins？" |
| 发布频率 | 生成的流程不匹配 | "发布频率是每天还是每季度？" |
| 合规要求 | 可能遗漏安全要求 | "是否有安全合规要求？GDPR/等保？" |

**应该有的流程：**
```
用户: "帮我生成 Harness"
  ↓
需求澄清对话:
  "这是新项目还是已有项目？"
  "团队规模多大？"
  "对代码质量的要求？（严格/标准/宽松）"
  "使用什么 CI 平台？"
  "有安全合规要求吗？"
  ↓
基于澄清结果，智能推荐:
  "根据您的需求，我推荐以下配置：
   - 质量门禁: 标准级别
   - CI 平台: GitHub Actions
   - 检查频率: 每次提交
   是否需要调整？"
  ↓
用户确认后生成
```

### 用户故事 2：已有代码库维护者 - 缺少确认环节

**当前流程（过于自动化）：**
```
用户: "帮我给这个项目生成 Harness"
  ↓
自动扫描项目
  ↓
自动推断特征
  ↓
自动生成系统
```

**问题：缺少关键确认**

| 缺失的确认 | 风险 | 应该确认的问题 |
|-----------|------|---------------|
| 推断准确性 | 生成的系统可能不符合实际 | "我识别到这是 Python + FastAPI 项目，对吗？" |
| 存量代码 | 生成的规范可能与存量代码冲突 | "项目中已有 500 个文件不符合规范，如何处理？" |
| 特殊约定 | 可能遗漏项目特殊约定 | "项目中是否有特殊的编码约定？" |
| 优先级 | 不知道用户最关心什么 | "您最关心质量的哪些方面？测试覆盖率？代码风格？" |

**应该有的流程：**
```
用户: "帮我给这个项目生成 Harness"
  ↓
扫描 + 推断:
  "我识别到以下项目特征：
   - 语言: Python
   - 框架: FastAPI
   - 包管理器: PDM
   - 测试框架: pytest
   
   是否正确？有需要补充或修正的吗？"
  ↓
用户确认/修正
  ↓
需求澄清:
  "扫描发现 500 个文件不符合 PEP8 规范。
   您希望：
   A. 立即全部修复
   B. 渐进式收紧（3个月）
   C. 暂不处理，只对新代码强制
   
   请选择或告诉我您的想法。"
  ↓
基于用户选择生成
```

### 用户故事 3：团队 Tech Lead - 缺少协作需求

**当前流程（过于自动化）：**
```
用户: "帮我管理多个项目的 Harness"
  ↓
自动创建 Registry
  ↓
自动生成配置
```

**问题：缺少团队协作需求澄清**

| 缺失的需求 | 影响 | 应该澄清的问题 |
|-----------|------|---------------|
| 权限管理 | 不知道谁可以修改什么 | "团队成员的权限如何划分？" |
| 版本策略 | 不知道何时升级规范 | "规范更新的流程是什么？需要审批吗？" |
| 共享程度 | 不知道项目间如何共享 | "哪些规范是所有项目共享的？" |
| 审计要求 | 可能遗漏合规要求 | "需要操作审计日志吗？" |

**应该有的流程：**
```
用户: "帮我管理多个项目的 Harness"
  ↓
需求澄清:
  "您管理多少个项目？"
  "团队有多少人？"
  "规范更新需要审批吗？"
  "哪些规范应该跨项目共享？"
  ↓
基于需求设计:
  "我建议以下结构：
   - 共享规范: 代码风格、安全要求
   - 项目特定: 技术栈相关的检查
   - 权限: Tech Lead 可修改共享规范，开发者可修改项目规范
   
   这样可以吗？"
  ↓
用户确认后创建
```

### 用户故事 4：CI/CD 工程师 - 缺少集成需求

**当前流程（过于自动化）：**
```
用户: "帮我生成 CI 配置"
  ↓
自动生成 GitHub Actions
```

**问题：缺少集成需求澄清**

| 缺失的需求 | 影响 | 应该澄清的问题 |
|-----------|------|---------------|
| CI 平台 | 可能生成不支持的平台 | "使用什么 CI 平台？GitHub Actions/GitLab CI/Jenkins？" |
| 触发条件 | 可能遗漏触发条件 | "需要在什么情况下触发检查？PR/Merge/Push？" |
| 通知方式 | 不知道如何通知 | "检查失败如何通知？Slack/邮件/评论？" |
| 缓存策略 | 不知道是否需要缓存 | "是否需要配置构建缓存？" |

**应该有的流程：**
```
用户: "帮我生成 CI 配置"
  ↓
需求澄清:
  "使用什么 CI 平台？"
  "在什么情况下触发？（PR/Merge/Push/Schedule）"
  "检查失败如何通知？"
  "需要配置构建缓存吗？"
  ↓
基于需求生成:
  "根据您的需求，我生成了以下 CI 配置：
   - 平台: GitHub Actions
   - 触发: PR 到 main 分支
   - 通知: Slack 通知到 #engineering 频道
   - 缓存: 启用 pip 缓存
   
   请确认或告诉我需要调整的地方。"
  ↓
用户确认后生成
```

## "自动化"不足总结

### 1. 需求澄清不足

| 问题 | 表现 | 影响 |
|------|------|------|
| **缺少主动询问** | 直接假设用户需求 | 生成的系统可能不符合预期 |
| **缺少优先级确认** | 不知道用户最关心什么 | 生成的内容可能主次不分 |
| **缺少上下文理解** | 不了解用户背景 | 生成的建议可能不切实际 |

### 2. 确认环节不足

| 问题 | 表现 | 影响 |
|------|------|------|
| **推断不确认** | 自动使用推断结果 | 可能基于错误的假设 |
| **选项不提供** | 只有一种选择 | 用户没有自主权 |
| **反馈不收集** | 生成后就结束 | 不知道是否满足需求 |

### 3. 迭代优化不足

| 问题 | 表现 | 影响 |
|------|------|------|
| **一次性生成** | 生成后不再优化 | 不能适应需求变化 |
| **学习不积累** | 不记住用户偏好 | 每次都要重新说明 |
| **历史不追溯** | 不知道之前的决策 | 无法解释为什么这样配置 |

## 优化建议

### P0：增加需求澄清对话

在每个 Skill 开始时，增加需求澄清对话：

```yaml
# harness-architect 需求澄清
clarification_questions:
  - question: "这是新项目还是已有项目？"
    options: ["新项目", "已有项目", "重构项目"]
    impact: 决定使用 Greenfield/Brownfield 模式
    
  - question: "团队规模多大？"
    options: ["个人项目", "小团队(<5人)", "中团队(5-20人)", "大团队(20+人)"]
    impact: 决定生成的规范严格程度
    
  - question: "对代码质量的要求？"
    options: ["严格（大厂标准）", "标准", "宽松（快速迭代）"]
    impact: 决定质量门禁阈值
    
  - question: "使用什么 CI 平台？"
    options: ["GitHub Actions", "GitLab CI", "Jenkins", "其他"]
    impact: 决定生成的 CI 配置格式
```

### P1：增加确认环节

在关键决策点增加确认：

```yaml
# 推断结果确认
confirmation_points:
  - point: "项目特征识别完成"
    message: |
      我识别到以下项目特征：
      - 语言: {language}
      - 框架: {framework}
      - 包管理器: {package_manager}
      
      是否正确？有需要补充或修正的吗？
    wait_for_user: true
    
  - point: "存量代码问题发现"
    message: |
      扫描发现 {count} 个文件不符合规范。
      您希望如何处理？
      A. 立即全部修复
      B. 渐进式收紧（3个月）
      C. 暂不处理，只对新代码强制
    wait_for_user: true
    options: ["立即修复", "渐进收紧", "只管新代码"]
```

### P2：增加迭代优化

支持生成后的迭代优化：

```yaml
# 迭代优化支持
iteration_support:
  - command: "/harness-adjust"
    description: "调整生成的配置"
    example: "/harness-adjust 测试覆盖率要求改为 70%"
    
  - command: "/harness-add"
    description: "添加新的检查项"
    example: "/harness-add 添加安全扫描检查"
    
  - command: "/harness-remove"
    description: "移除不需要的检查项"
    example: "/harness-remove 移除类型检查"
```

## 重新定义"自动化"

### 真正的自动化应该是：

| 类型 | 任务 | 处理方式 |
|------|------|----------|
| **机械性** | 文件扫描、配置生成、格式转换 | ✅ 自动执行 |
| **决策性** | 需求澄清、优先级确定、方案选择 | ❌ 需要用户参与 |
| **创造性** | 方案设计、最佳实践推荐 | ⚠️ AI 建议 + 用户确认 |

### 优化后的流程：

```
用户发起请求
  ↓
【对话】需求澄清（新增）
  ↓
【自动】扫描项目特征
  ↓
【对话】确认推断结果（新增）
  ↓
【对话】选择方案（新增）
  ↓
【自动】生成定制系统
  ↓
【对话】确认生成结果（新增）
  ↓
【自动】保存配置
  ↓
【对话】收集反馈（新增）
```

## 总结

当前 harness skills 的"自动化"不足在于：

1. **过度自动化** - 忽略了需求澄清的重要性
2. **单向输出** - 不与用户交流确认
3. **缺乏迭代** - 生成后不支持优化

**核心问题**：自动化应该用于**机械性**任务，而**决策性**任务需要与用户交流。

**优化方向**：
- 增加需求澄清对话
- 增加关键确认环节
- 增加迭代优化支持
- 让自动化服务于用户，而非替代用户
