# Your Harness 项目开发路线图

## 核心目标

**从"模板匹配"转型为"元能力驱动"**

当前问题：依赖预定义元模板，无法覆盖所有场景
目标状态：AI 动态理解项目，智能生成定制化流程

---

## 阶段 0：当前状态（v4.6.0）

### 架构

```
项目特征 → 匹配预定义模板 → 生成流程
                ↓
         没有匹配的模板 → 使用通用模板
```

### 问题

- 元模板是静态的，无法覆盖所有场景
- 模板质量决定生成质量
- 维护成本高（每增加一种项目类型，需要添加模板）

### 现有资产

- 15 个项目模式定义
- 8 个元模板
- 2 个脚本模板
- 知识库（project-patterns.yaml, feature-to-product-mapping.yaml）

---

## 阶段 1：元能力核心（v5.0.0）

### 目标

实现"元能力"，让 AI 能够动态生成流程，而非从预定义模板选择。

### 核心变更

#### 1.1 重构 harness-designer.md

**从"模板匹配"改为"AI 动态生成"**

```yaml
# 旧逻辑
Step 1: 匹配项目模式
Step 2: 选择对应模板
Step 3: 填充变量

# 新逻辑
Step 1: AI 分析项目特征
Step 2: AI 推理需要的开发阶段
Step 3: AI 生成定制化流程
Step 4: AI 组合已有知识（如果适用）
```

#### 1.2 定义元能力

创建 `meta-capabilities/` 目录，定义 5 个核心元能力：

```
meta-capabilities/
├── 01-project-analysis.md      # 项目特征提取能力
├── 02-stage-reasoning.md       # 开发阶段推理能力
├── 03-flow-generation.md       # 流程结构生成能力
├── 04-knowledge-composition.md # 知识组合能力
└── 05-learning-feedback.md     # 学习反馈能力
```

#### 1.3 元能力 1：项目特征提取

```yaml
capability: project-analysis

description: |
  教 AI 如何分析项目，提取以下维度的特征：
  
  1. 技术维度
    - 语言、框架、工具链
    - 包管理器、构建系统
    - 测试框架、CI/CD
    
  2. 架构维度
    - 单体/微服务/Monorepo
    - 前后端分离/全栈
    - 依赖关系复杂度
    
  3. 业务维度
    - AI/LLM 相关
    - 数据密集型
    - 实时性要求
    - 安全敏感度
    
  4. 团队维度
    - 代码风格
    - 文档完善度
    - 现有规范

input: 项目代码和配置
output: 结构化的项目特征描述
```

#### 1.4 元能力 2：开发阶段推理

```yaml
capability: stage-reasoning

description: |
  教 AI 如何根据项目特征，推理出需要哪些开发阶段。
  
  推理规则（不是针对特定项目的，是通用规则）：
  
  规则 1：外部集成规则
    if 项目涉及"调用外部系统/API"
    then 需要阶段：API 分析 → 接口设计 → 实现 → 兼容性测试
    
  规则 2：数据处理规则
    if 项目涉及"数据存储/处理"
    then 需要阶段：数据模型设计 → 数据库设计 → 实现 → 数据测试
    
  规则 3：用户交互规则
    if 项目涉及"用户界面"
    then 需要阶段：需求分析 → UI 设计 → 组件开发 → 交互测试
    
  规则 4：安全敏感规则
    if 项目涉及"认证/授权/敏感数据"
    then 需要阶段：安全设计 → 实现 → 安全审查 → 渗透测试
    
  规则 5：性能要求规则
    if 项目涉及"高并发/实时性"
    then 需要阶段：性能分析 → 架构设计 → 实现 → 性能测试
    
  ...更多通用规则

input: 项目特征
output: 需要的开发阶段列表 + 每阶段的关注点
```

#### 1.5 元能力 3：流程结构生成

```yaml
capability: flow-generation

description: |
  教 AI 如何把推理结果转化为具体的流程定义。
  
  流程骨架（所有项目通用）：
  
  阶段 0: 需求理解
    - 目标：理解要做什么
    - 输出：需求文档
    - 闸门：需求确认
    
  阶段 1: 任务分解
    - 目标：拆解为可执行任务
    - 输出：任务列表
    - 闸门：任务确认
    
  阶段 2: 设计
    - 目标：确定怎么做
    - 输出：设计文档
    - 闸门：设计评审
    
  阶段 3: 实现
    - 目标：写出代码
    - 输出：代码 + 测试
    - 闸门：代码审查
    
  阶段 4: 验证
    - 目标：确保质量
    - 输出：测试报告
    - 闸门：质量检查
    
  阶段 5: 上线
    - 目标：安全发布
    - 输出：部署配置
    - 闸门：上线审批
  
  AI 的工作：
  1. 填充每个阶段的具体内容（基于项目特征）
  2. 添加项目特定的检查项
  3. 引用合适的 Skill

input: 开发阶段列表 + 关注点
output: 完整的流程定义（SKILL.md + templates）
```

#### 1.6 元能力 4：知识组合

```yaml
capability: knowledge-composition

description: |
  教 AI 如何组合已有知识，处理混合场景。
  
  组合策略：
  
  策略 1：特征分解
    将复杂项目分解为多个简单特征
    例如：AI Agent 项目 = LLM 集成 + API 设计 + 数据存储
    
  策略 2：知识检索
    从知识库中检索相关经验
    如果有类似项目的成功案例，参考其流程
    
  策略 3：模式匹配
    识别项目匹配的已知模式
    如果匹配度高，优先使用已知模式的流程
    
  策略 4：创新生成
    对于完全未知的部分
    基于通用原则和最佳实践生成

input: 项目特征 + 知识库
output: 组合后的流程定义
```

#### 1.7 元能力 5：学习反馈

```yaml
capability: learning-feedback

description: |
  教 AI 如何从每次生成中学习，持续改进。
  
  学习流程：
  
  1. 生成后询问：
    - "生成的流程是否满足需求？"
    - "有哪些地方需要调整？"
    
  2. 记录反馈：
    - 哪些推理是准确的
    - 哪些推理需要修正
    - 用户的修改建议
    
  3. 更新知识库：
    - 添加新的项目特征 → 流程映射
    - 修正错误的推理规则
    - 提升成功的模式为"最佳实践"

input: 生成结果 + 用户反馈
output: 更新后的知识库
```

## 阶段 2：混合模式（v5.1.0）

### 目标

实现"元模板 + 元能力"的混合模式，兼顾效率和覆盖度。

### 核心变更

#### 2.1 智能路由

```yaml
# 在 harness-designer.md 中添加路由逻辑

routing_strategy:
  priority_1: 元模板精确匹配
    condition: "项目特征完全匹配已知模式"
    action: "使用预定义的元模板"
    benefit: "快速生成，质量保证"
    
  priority_2: 元模板部分匹配
    condition: "项目特征部分匹配已知模式"
    action: "使用元模板 + 元能力补充"
    benefit: "利用已有知识 + 处理新特征"
    
  priority_3: 元能力生成
    condition: "项目特征不匹配任何已知模式"
    action: "完全使用元能力动态生成"
    benefit: "覆盖未知场景"
    
  priority_4: 通用流程
    condition: "元能力也无法生成满意结果"
    action: "使用最基础的通用流程"
    benefit: "兜底，确保总是有输出"
```

#### 2.2 元模板优化

将现有的元模板改造为"可组合的元素库"：

```
meta-templates/
├── elements/                    # 可组合的元素
│   ├── stages/                  # 阶段定义
│   │   ├── api-analysis.md      # API 分析阶段
│   │   ├── security-review.md   # 安全审查阶段
│   │   ├── performance-test.md  # 性能测试阶段
│   │   └── ...
│   ├── checks/                  # 检查项
│   │   ├── api-compatibility.md
│   │   ├── cost-estimation.md
│   │   └── ...
│   └── gates/                   # 闸门定义
│       ├── code-review.md
│       ├── test-coverage.md
│       └── ...
│
└── combinations/                # 预定义的组合（现有模板）
    ├── TEMPLATE-llm-integration.md
    ├── TEMPLATE-microservice.md
    └── ...
```

#### 2.3 组合引擎

```yaml
capability: combination-engine

description: |
  根据项目特征，智能组合元素：
  
  1. 分析项目需要哪些阶段元素
  2. 分析项目需要哪些检查项
  3. 分析项目需要哪些闸门
  4. 组合为完整的流程定义
  
  组合规则：
  - 相同类型的元素只保留一个
  - 调整元素的顺序
  - 处理元素之间的依赖关系
```

#### 3.2 学习流程

```yaml
# 每次生成后的学习流程

after_generation:
  1. 收集生成结果
     - 生成的流程定义
     - 用户的使用反馈
     - 用户的修改记录
     
  2. 分析结果
     - 识别项目特征
     - 评估生成质量
     - 提取成功/失败模式
     
  3. 更新知识库
     - 添加新的特征 → 流程映射
     - 修正错误的推理规则
     - 如果质量高，提升为最佳实践
     
  4. 通知用户
     - "已学习本次生成，感谢使用"
     - "如有建议，欢迎反馈"
```

#### 3.3 知识提升机制

```yaml
# 从学习记录提升为最佳实践的机制

promotion_rules:
  1. 使用次数
     - 如果某个学习记录被参考超过 N 次
     - 且用户满意度高
     - 提升为"最佳实践模板"
     
  2. 质量评分
     - 如果某个学习记录的质量评分超过阈值
     - 提升为"最佳实践模板"
     
  3. 人工审核
     - 如果用户标记为"推荐"
     - 提升为"最佳实践模板"
```

## 阶段 4：智能优化（v6.0.0）

### 目标

实现持续的智能优化，让 Your Harness 越用越聪明。

### 核心变更

#### 4.1 推理规则优化

```yaml
# 基于历史数据优化推理规则

optimization:
  1. 分析历史生成结果
     - 哪些推理规则命中率高
     - 哪些推理规则经常出错
     
  2. 优化规则
     - 提高命中率高的规则的权重
     - 修正经常出错的规则
     - 添加新的规则
     
  3. A/B 测试
     - 新规则 vs 旧规则
     - 评估改进效果
```

#### 4.2 个性化适配

```yaml
# 根据用户/团队的偏好进行个性化

personalization:
  1. 学习用户偏好
     - 用户喜欢什么样的流程
     - 用户常用的 Skill
     - 用户的审查标准
     
  2. 生成个性化流程
     - 基于用户偏好的流程风格
     - 基于用户习惯的闸门设置
     
  3. 团队共享
     - 团队可以共享偏好设置
     - 团队可以共享最佳实践
```

#### 4.3 预测性建议

```yaml
# 主动预测用户需求

prediction:
  1. 分析当前项目状态
     - 当前在哪个阶段
     - 下一个阶段是什么
     - 可能遇到的问题
     
  2. 提前准备
     - 预生成下一个阶段的模板
     - 提前加载可能需要的 Skill
     
  3. 主动建议
     - "根据历史经验，这个阶段常见问题有..."
     - "建议在...之前检查..."
```

## 总结

这个路线图的核心思想是：

1. **从"模板匹配"到"元能力驱动"**：让 AI 理解如何生成流程，而非记忆模板
2. **混合模式**：元模板（效率）+ 元能力（覆盖）+ 学习（改进）
3. **持续优化**：每次使用都是一次学习机会，系统越用越聪明

最终目标是让 Your Harness 成为一个"能够自我进化的开发流程生成器"。
